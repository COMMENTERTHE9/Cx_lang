## Landed today

All 4 commits landed on `origin/submain`. None have been merged to main.

- **non-exhaustive `when` is a semantic error** (`5437af7`, tracker #027) — CONFIRMED. A `when` block with no catch-all `_` arm now fails at analysis time with "non-exhaustive `when`: add a `_` catch-all arm to handle the remaining cases." The rule is lenient (TBool-exempt): a match covering true/false/unknown counts as exhaustive without `_`, preserving the canonical Cx three-state idiom. Enum-group and super-group patterns already lower to `Catchall`, so they pass. Bare enum-variant matches and open-domain patterns (numeric, range, string) require a `_`. Full static enum exhaustiveness is deferred to 0.3. The check fires at both the statement-when and expression-when sites in `semantic.rs`. Migrated one fixture (`t22_enum_basic` gained a `_` for its 4-variant Direction match). One new expected-fail fixture.

- **`if <unknown>` is a runtime error directing to `when`** (`b3faea9`, tracker #026) — CONFIRMED. An `if` whose condition evaluates to `TBool(2)` (unknown) or `Unknown(_)` used to silently take the else branch — the `_ => false` fall-through in the IfElse condition check. This collapsed the third state that TBool exists to express. Now it returns `RuntimeError::UnknownCondition` with the message: "`if` condition is unknown; an unknown TBool can't choose a branch — use `when` to handle true, false, and unknown explicitly." The check is at the runtime eval site (not semantic) because unknown-ness can arise from runtime-only expressions like `a && ?`. New variant and renderer added to `types.rs` and `diagnostics.rs`. One new expected-fail fixture. Aligns interpreter behavior with the JIT, which already rejected unknown-typed conditions at lowering.

- **enums in function signatures** (`b8bcf1d`, tracker #019) — CONFIRMED. The parser emits all bare-name type annotations as `Type::Struct(name)`, so an enum used as a parameter type, return type, or typed binding failed to unify with enum values (inferred as `SemanticType::Enum`), producing the confusing "expected Light, got enum Light" mismatch where both names looked identical. The fix adds an enum-normalization pre-pass that rewrites `Type::Struct(name) → Type::Enum(name)` for every declared enum, recursing through all nested statement bodies (function bodies, impl blocks, loops, if/when arms). The parser is untouched. A one-line cosmetic fix in `type_name` drops the internal `enum ` prefix from user-facing messages. 148 lines of new code in `semantic.rs`. Three new fixtures: `t_enum_param`, `t_enum_return`, and `t_enum_state_machine` (the state-machine pattern that couldn't be written before this fix). Cross-module imported enum annotations remain `Struct` for now — a 0.3 follow-up.

- **error on non-variable string interpolation** (`e16a059`, tracker #038) — CONFIRMED. A `{...}` segment in an interpolated string that didn't resolve to a bound variable was silently re-emitted as literal text. A typo like `{toatl}` or a non-variable expression like `{fib(i)}` appeared verbatim in output instead of erroring — silent wrong output. `expand_interpolation` now returns `Result` and errors with `RuntimeError::BadInterpolation`, with a mode-specific hint: undefined name → "no variable `toatl` in scope"; non-variable expression → "supports bare variables only; compute `{fib(i)}` into a variable first." `print_value` and `print_value_inline` now propagate the Result at the two `call.rs` print sites. New `is_bare_identifier` helper distinguishes the two failure modes. Known limitation: diagnostic reports line 1, not the print call's real line (call-position threading is a separate cross-cutting item). Two new expected-fail fixtures.

## Decisions and direction

Today's commits reinforce the shift-left-on-error-quality direction established yesterday. Three of four commits make the compiler stricter about things that were previously silent:

**TBool safety tightened.** Both `when` (#027) and `if` (#026) now enforce TBool's three-state contract. A `when` without a catch-all is caught at compile time. An `if` on unknown is caught at runtime. Together they close the two routes by which an unknown value could silently collapse to a boolean decision. This is a consequential design choice for Cx — it makes TBool's unknown state a first-class concern that programs must handle explicitly, not a value that degrades to false.

**Enum type annotations fixed.** The parser's ambiguity between enum and struct names in annotation position (#019) was a latent bug since enums were introduced. The normalization pre-pass approach keeps the parser clean (no grammar change) and handles all nesting depths. The decision to drop the `enum ` prefix from user-facing type names unifies how all named types render. Cross-module enum annotation resolution is explicitly scoped as a 0.3 item.

**String interpolation hardened.** The choice to error on unresolvable `{...}` instead of treating it as literal text (#038) closes a silent-wrong-output path. The explicit deferral of full expression interpolation to 0.3 (with a clear error and workaround message today) is a pragmatic scope boundary.
