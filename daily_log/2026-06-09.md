## Decisions and direction

CR#4 completed a deliberate architectural consolidation: **unifying all literal width checking through a single semantic-level function**. Before CR#4, `check_literal_fits` operated on AST `Expr` nodes and `check_semantic_num_fits` operated on `SemanticExpr` nodes — structurally identical logic duplicated at two abstraction levels. CR#4 deleted `check_literal_fits` and routed every width-check site through `check_semantic_num_fits`, which can now also descend into branch tails.

The four commits form a complete tracker:
- CR#1: generic type args → range check (b8e92fb)
- CR#2: array elements in struct fields and call args → range check (7337b61)
- CR#3: return values → range check (79a1bbd)
- CR#4: if/when branch tails → range check + delete `check_literal_fits` (cbba042)

Each commit added one `analyze_expr_with_elem_hint` or `analyze_returned_expr` helper, converging toward a smaller set of analyzed entry points. The remaining non-literal narrowing gap (computed expressions like `a + b` producing out-of-range values) is explicitly deferred — it depends on cast syntax, noted in CR#4's commit message as "tracked separately in the design queue."
