## Landed today

**String interpolation + read/input built-ins — CONFIRMED** (775b30d)
`expand_interpolation` method added to the runtime. `{varname}` in string literals is expanded at print time; unknown variables pass through literally. `read(var)` reads a line from stdin into a variable, `input("prompt", var)` prints a prompt then reads. Test t60_read_basic.cx updated to use interpolation syntax. Two hard blockers closed in one commit: string interpolation and stdin input.

**Generic structs Phase 1 — CONFIRMED** (9a6d3d8)
`type_params` added to `Stmt::StructDef` and `SemanticStmt::StructDef`. Parser handles `struct Foo<T> { field: T }` syntax. Analyzer sets `current_type_params` during field analysis so `T` resolves correctly. New `struct_type_params` registry in the `Analyzer`. Tests t61 (generic struct basic) and t62 (generic struct f64) both passing. Also fixed run_matrix.sh CRLF line endings.

**Generic structs Phase 2 — CONFIRMED** (532d17e)
`AstValue::StructInstance` now carries `Vec<Type>` type args. Parser accepts `Pair<t32> { ... }` instantiation syntax. All `StructInstance` match sites updated for the three-field form. Test t63 (generic struct explicit type args) passing. Known gaps remain: type args in variable type annotations (`p: Pair<t32>`) not yet parsed, and generic field type checking is not enforced at the semantic layer.

**Promote print/printn to functions — CONFIRMED** (4e22e28)
`KeywordPrint` and `KeywordPrintInline` removed from the lexer. Print statement combinators removed from the parser. `print` and `printn` now parse as `Expr::Call` like all other functions. The `call_arg` parser was fixed to handle nested calls and method calls in arguments. `printn` added as a built-in in `call_semantic_func`. The IO surface is now fully consistent — print, printn, read, and input are all functions. The print-as-function refactor also resolved the t42 TypeParam-vs-Struct ambiguity: removing print keywords from the lexer eliminated the parsing path that caused the collision. t42's `.expected_fail` marker was removed. Net result: 47 lines of parser combinators deleted, 11 lines of runtime added.

**Delete dead enum group infrastructure — CONFIRMED** (2de3e1a)
`EnumRuntimeInfo` struct, the `enums` field on `RunTime`, and `super_group_handler_index` all deleted. These were write-only — when-group matching works entirely through `SemanticWhenPattern` now, so the runtime struct was dead weight. 37 lines removed from runtime.rs, 8 lines from main.rs.

## Decisions and direction

**IO surface unified.** With print/printn promoted to functions alongside read and input, the entire IO surface now follows the same calling convention. This was explicitly called out as a prerequisite before the backend locks calling convention, and it's now done. The lexer no longer has any IO-specific keywords.

**Generic structs are real but incomplete.** Phase 1+2 landed definition and instantiation, but the commit messages explicitly call out two known gaps: type args in variable declarations and generic field type checking. These are tracked in the roadmap under Active.

**t42 ambiguity resolved unexpectedly.** The TypeParam-vs-Struct collision in nested generic calls — tracked as a known gap since generics v2 — turned out to be a side effect of print being a keyword. Once print parsed as a regular call, the ambiguity path disappeared. This is a good example of how cleaning up one system can fix bugs in another.

**Dead code continues to shrink.** The enum group runtime infrastructure deletion follows yesterday's 790-line AST interpreter removal. The codebase is getting leaner as the semantic path takes over.
