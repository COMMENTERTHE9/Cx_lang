**D1.1b — Fix array index element type in read-expressions** (`aadbd55`, 14:50 -0400) — CONFIRMED

Live interpreter correctness bug found during benchmark work. `Expr::Index` in read position hardcoded `ty: SemanticType::Unknown` even though the array base carries a known element type. This caused array-element arithmetic at sub-64-bit widths to skip wrapping, producing wrong answers:

```
a:[k] + a:[k]  (t8 100+100)   200 → -56  (now matches plain t8 vars)
x: t8 = a:[k] + a:[k]         200 → -56  (out-of-range no longer leaks)
```

Fix: ~5 lines in `semantic.rs`, recover element type from `Array(_, elem)` base in the Index read arm. Six new fixtures pin t8/t16 wrapping, typed binding narrowing, and return-value wrapping. Benchmark `bench_array_loop` dropped its typed-local workaround and now runs the natural form.

A methodology finding also landed in `bench/BASELINE.md`: absolute wall-times are not cross-session comparable (thermal state caused a uniform ~2.4x swing on untouched programs). Standing rule revised to within-session A/B comparison.

Matrix: 272/0. Parity: 171 PASS / 101 SKIP / 0 PARITY_FAIL (+6 new fixtures, all lower and agree interp==JIT).

**D1.1 — Unify integer-width facts** (`a5c930d`, 15:13 -0400) — CONFIRMED

The first true "judgment extraction" of the D1 arc. "What fits a tN" lived in three hand-synced copies across `semantic.rs` (range-check bounds), `runtime.rs` (narrowing casts), and `lower.rs` (IR range validation). New leaf module `frontend/int_facts.rs` is the single source of truth, exposing `IntWidth {W8..W128}`, `IntFacts {min, max, name}`, and a `truncate(i128)` method. Each consumer maps its own type into `IntWidth` and reads the one table.

Design decisions:
- Pure-data leaf with no dependencies on SemanticType or IrType — dependency points only into it, no cycle.
- Bool excluded from the table (only IR needs its 0/1 bound, retained as a one-line local arm).
- `bits` dropped from the shape — no consumer reads it.

7 files changed, 129 insertions, 26 deletions. Behavior-identical by construction — all gate counts and diagnostic messages frozen. Matrix: 272/0.

## Decisions and direction

Today marked a deliberate pivot from the convergence/merge track to a soundness/quality track. Instead of merging CR#1–4 to main or resuming backend work, the developer opened the D1 audit arc — a systematic pass through the codebase to find and fix latent correctness issues and extract duplicated logic.

The arc has a clear internal structure:
- **D1.0**: make the unmeasured measurable (pin crash classes so they cannot regress silently)
- **perf-rider**: establish a performance baseline before refactoring begins
- **D1.1b**: fix a real bug surfaced by the benchmark work (array index types)
- **D1.1**: extract the first judgment (integer-width facts → single table)

The D1.1 extraction follows the same architectural pattern as the earlier `builtins.rs` (#008) — a pure-data leaf module that multiple layers consult without creating circular dependencies. This appears to be a deliberate, repeating design strategy for Cx: when a fact is used in 3+ places, it gets a leaf module with a neutral key enum.

The benchmark methodology finding (absolute wall-times not cross-session comparable due to thermal state) was documented immediately and the standing rule revised — a sign that the project treats methodology corrections as first-class findings, not afterthoughts.
