- **CR#1: range-check fields through explicit generic type args** (`b8e92fb`, `origin/submain`) — CONFIRMED. `Pair<t8> { b: 300 }` silently accepted 300 because explicit generic type arguments were discarded — the `_type_args` in the StructInstance arm were unused, so field literals validated against the raw `TypeParam("T")`, making `check_literal_fits` a no-op. Fix: when `type_args` is non-empty, build a positional param→arg substitution map and apply it to each field's declared type before range-checking. Also adds arity checking in both directions (extra or too-few type args now error). 13 files, 6 new fixtures.

- **CR#2: range-check array elements in struct fields and call args** (`7337b61`, `origin/submain`) — CONFIRMED. Array literals only range-checked elements at standalone declarations. Struct fields (`items: [3: t8] = [1, 2, 300]`) and function arguments silently accepted out-of-range values because the element-type hint was never set at those sites. Fix: a shared `analyze_expr_with_elem_hint` helper now delivers the element-type hint at struct-field and call-argument sites, DRYing the typed-declaration path into it. Composes with CR#1 through generic substitution. Also surfaced a pre-existing Cranelift codegen segfault — negative elements in struct fixed-size array fields corrupt adjacent stack memory (marked `.jit_known_unsound`). 18 files, 8 new fixtures.

- **CR#3: range-check return values at declared width** (`79a1bbd`, `origin/submain`) — CONFIRMED. `fnc: t8 f() { 300 }` silently returned 44 — the trailing-return path went straight to `insert_cast_if_needed` (which wraps) with no range check. Silent-wrong-value, the worst class. Fix: a shared `analyze_returned_expr` chokepoint now runs the same sequence a typed declaration runs — array-element hint (CR#2), bare-literal width check (#028/#037), type-compatibility gate, then the cast — and both return forms (explicit `return` and trailing expression) route through it. Also upgrades the explicit-return path to include the array-element check. 17 files, 8 new fixtures.

## Decisions and direction

**The #028/#037 rule is now fully closed across five surfaces.** The original rule (out-of-range integer literals must reject at semantic time, both signs, at the declared width) was established in the v0.1 era for declarations. CR#1–3 extended it to generic type args, array elements in struct fields and call args, and return position. The only remaining gap the commits identify is if/when branches (`{ if c { 300 } else { 0 } }`), tracked as CR#4 — a check-descent issue, not a surface-specific one.

**Implicit narrowing is a language design question, not a bug fix.** CR#3 explicitly scopes out implicit non-literal narrowing (`x: t8 = a` for `a: t64`), noting that Cx has no explicit cast syntax and rejecting it would strand legitimate truncation. This is queued as a language-design ruling (cast syntax first, then reject-implicit-narrowing). This is a deliberate decision to leave wrapping behavior intact for variable-to-variable assignments until the language has a way to express the intent.

**Cranelift negative-element struct array segfault surfaced.** CR#2 exposed a pre-existing JIT codegen bug: storing a negative value in a struct's fixed-size array field corrupts adjacent stack memory on teardown. Positive arrays work fine. Marked `.jit_known_unsound` (the #003/#032 convention). The interpreter is correct and serves as the reference.
