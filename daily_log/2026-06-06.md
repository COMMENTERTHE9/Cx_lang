- **CR#1: range-check fields through explicit generic type args** (`b8e92fb`, `origin/submain`, 15:22 EDT) — CONFIRMED. Found by CodeRabbit during the release PR review. `Pair<t8> { b: 300 }` silently accepted 300 because explicit generic type arguments were discarded — the `_type_args` in the `StructInstance` arm of `semantic.rs` were unused, so field literals validated against raw `TypeParam("T")`, making `check_literal_fits` a no-op. Fix: when `type_args` is non-empty, check arity in both directions (extra args silently dropped by zip, too few leave params unsubstituted), build a positional param-to-arg substitution map via `substitute_type_params`, and apply it to each field's declared type before range-checking. Added 6 new fixtures. Matrix: 236/0.

- **CR#2: range-check array elements in struct fields and call args** (`7337b61`, `origin/submain`, 16:45 EDT) — CONFIRMED. Array literals only range-checked elements at standalone declarations (`x: [3: t8] = [1, 2, 300]` rejects). Struct fields (`items: [3: t8]` initialized with `[1, 2, 300]`) and function arguments silently accepted 300 because the element-type hint was never set at those sites. Fix: a new `analyze_expr_with_elem_hint` helper in `semantic.rs` delivers the hint at struct-field and call-argument sites, consolidating the typed-declaration path into it. Composes with CR#1 — `Box<t8> { items: [3: T] }` now resolves to `[3: t8]` and range-checks elements. Also surfaced a pre-existing Cranelift codegen segfault: negative values in struct fixed-size array fields corrupt adjacent stack memory (marked `.jit_known_unsound`). Added 8 new fixtures. Matrix: 244/0.

- **CR#3: range-check return values at declared width** (`79a1bbd`, `origin/submain`, 18:11 EDT) — CONFIRMED. `fnc: t8 f() { 300 }` silently returned 44 — the trailing-return path went straight to `insert_cast_if_needed` (which wraps) with no range check. Silent-wrong-value, the worst class of bug. Fix: a shared `analyze_returned_expr` chokepoint in `semantic.rs` runs the same sequence a typed declaration runs — array-element hint (CR#2), bare-literal width check (#028/#037), type-compatibility gate, then cast — and both return forms (explicit `return expr` and trailing expression) route through it. Added 8 new fixtures. Matrix: 252/0.

## Decisions and direction

**The #028/#037 out-of-range literal rejection rule is now closed across five surfaces.** The original rule — out-of-range integer literals must reject at semantic time, both signs, at the declared width — was established for standalone declarations in the v0.1 era. CR#1-3 extended it to explicit generic type arguments, array elements in struct fields and call arguments, and return position (both trailing and explicit). The only remaining gap the commits identify is literals nested inside if/when branches (`{ if c { 300 } else { 0 } }`), tracked as CR#4 — a check-descent issue, not a surface-specific one.

**Implicit non-literal narrowing is deliberately left alone.** CR#3 explicitly scopes out implicit variable-to-variable narrowing (`x: t8 = a` for `a: t64`). Cx has no explicit cast syntax, so rejecting it would strand legitimate truncation with no escape hatch. This is queued as a language-design decision: add cast syntax first, then reject implicit narrowing.

**Cranelift negative-element struct array segfault documented.** CR#2 exposed a pre-existing JIT codegen bug where storing a negative value in a struct's fixed-size array field corrupts adjacent stack memory on teardown. Positive arrays work fine. Marked `.jit_known_unsound` per the #003/#032 convention. The interpreter is correct and serves as the reference.

**Runtime refactoring landed on main.** The v0.2.0 merge brought the runtime split (1725-line `runtime.rs` decomposed into `exec.rs`, `eval.rs`, `call.rs`, `ops.rs`, `scope.rs`, `print.rs`). This is a pure refactor with no behavior change, but it establishes the module structure for future runtime work.
