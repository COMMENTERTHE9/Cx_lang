**Enforce wrapping arithmetic — CONFIRMED** (`d93582b` on submain)

Fixed inconsistent overflow behavior in `src/runtime/runtime.rs`. Changed `saturating_add` → `wrapping_add`, `saturating_sub` → `wrapping_sub`. Added guards for `i128::MIN / -1` (was a latent panic) and `i128::MIN % -1`. All arithmetic now wraps consistently at i128 range. Width truncation to declared type still happens at assignment via `apply_numeric_cast`. This is a partial fix toward the hard blocker — arithmetic is now wrapping instead of saturating, but per-type-width wrapping at arithmetic time (e.g., t8 wrapping at 255 during addition, not just at assignment) is not yet implemented.

## Decisions and direction

**All compound type layouts are now locked for 0.1.** With struct, array, and enum layout joining the scalar layout from yesterday, the ABI doc covers every data type that exists in the language at 0.1 scope. The enum layout decision (tag-only u8, no payloads) is deliberately minimal — payload enums are explicitly post-0.1.

**Calling convention is locked.** Single return value, C ABI. The `copy_into` parameter semantics are deferred to post-0.1. This simplifies the backend considerably for 0.1 — every function call is a standard C-convention call.

**Backend is moving into control flow.** The jump from Phase 8 (data layout) to Phase 10 (loop lowering) in a single day signals that the backend is accelerating. With data layout locked and calling convention decided, the remaining backend work is control flow lowering (loops, conditionals) and codegen.

**Wrapping arithmetic is now consistent but not per-width.** The runtime fix ensures all arithmetic wraps (not saturates) at i128, with type-width truncation at assignment. Full per-type-width wrapping at arithmetic time is still an open item. This is an engineering tradeoff — the current approach is correct for most programs but could produce surprising intermediate values in expressions involving narrow types.
