## In progress / uncommitted work

The wonderful-saha worktree contains the most significant in-progress work. It is based on the submain branch (at 641cc39, Semantic Parity Phase 1) and has 782 uncommitted insertions across 9 files. This is Semantic Parity Phase 2 — the interpreter migration to SemanticProgram.

**What the uncommitted work does:**

- **`src/runtime/runtime.rs` (+627 lines):** A full semantic IR interpreter has been written. New methods `eval_semantic_expr`, `run_semantic_stmt`, and `call_semantic_func` implement expression evaluation and statement execution against `SemanticExpr` and `SemanticStmt` types instead of raw AST nodes. The RunTime struct gains a `semantic_funcs` registry (`HashMap<String, SemanticFunction>`) and a `register_semantic_func` method. This is the core of the migration — the interpreter can now run semantic IR directly.

- **`src/main.rs` (+83/-85 lines):** `run_with_interpreter` now accepts `SemanticProgram` instead of `Program`. The pre-pass that registers structs, functions, and impl blocks has been rewritten to work with `SemanticStmt` variants. Function registration bridges semantic and legacy registries to maintain copy/bleedback fallback behavior during the transition.

- **`src/frontend/ast.rs` (+28 lines):** New AST nodes added — `IfElse` and `WhileIn` statement variants, plus a `WhileInChain` struct for cursor iteration chains. These nodes formalize control flow constructs that were previously handled through different paths.

- **`src/frontend/semantic_types.rs` (+28 lines):** Corresponding `SemanticStmt::IfElse` and `SemanticStmt::WhileIn` variants added, plus `SemanticWhileInChain`. The semantic IR now has first-class representations for all control flow constructs.

- **`src/frontend/semantic.rs` (+68/-85 lines):** The analyzer's `semantic_type_from_decl` now takes a generic params slice, and `analyze_function` gains a type parameters argument. Duplicate StructDef handling removed. These changes prepare the analyzer for generics-aware semantic analysis.

- **`src/frontend/types.rs` (+20 lines), `src/frontend/diagnostics.rs` (+4 lines), `src/frontend/parser.rs` (+7/-1 lines), `src/ir/lower.rs` (+2 lines):** Supporting changes — type conversions, diagnostic helpers, parser adjustments, and stub lowering cases for the new AST variants.

This work is clearly incomplete but substantial. The semantic interpreter is written. The main.rs entry point has been rewired. But the bridge between old and new runtime paths is still being built — the copy/bleedback fallback pattern suggests this is a staged migration, not a clean swap.

## Decisions and direction

- **Staged interpreter migration rather than clean swap.** The wonderful-saha work maintains both the old AST function registry and the new semantic function registry simultaneously. `register_semantic_func` coexists with `register_func`. This is a practical choice — it lets the migration proceed incrementally while the old paths remain as fallback, reducing the risk of breaking the 45-test matrix during the transition.

- **IfElse and WhileIn promoted to first-class AST/semantic nodes.** Previously these constructs were handled through more general paths. Giving them dedicated AST and SemanticStmt variants makes the semantic IR complete for control flow — a prerequisite for the interpreter to run entirely off semantic IR.

- **Analyzer gains generic parameter awareness.** The `semantic_type_from_decl` signature change (adding `&[]` generic params) is groundwork for Generics v2 support in the semantic layer, even though this change is focused on interpreter migration.
