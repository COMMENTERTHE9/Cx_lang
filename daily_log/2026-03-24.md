## Landed today

**Macro system + `#![imports]` syntax — CONFIRMED** (`cda45f7`, merged via PR #17 `1cce577`)

This is a substantial commit touching 8 source files and adding 9 test files (22 files total, +210 lines / -13 lines). The work breaks down into six areas:

**Lexer** — Two new tokens: `MacroInnerOpen` for `#![` and `MacroOuterOpen` for `#[`. `PunctBracketClose` reused as the closing delimiter for both macro forms.

**Import block parsing** — New `ImportDecl` AST node with alias, path, and position. New `Stmt::ImportBlock` variant. The parser handles the full `#![imports]` block syntax:
```cx
#![imports]
math: use "std/math"
player: use "./player"
```
The `import_decl` and `import_block` combinators parse alias-colon-use-path declarations inside the block.

**Import semantic validation** — Duplicate alias rejection catches repeated import aliases. Registry path rejection enforces that only `./` (relative) and `std/` (stdlib) paths are allowed in v0.1. `std/` paths are accepted. This is validation only — actual file loading and module resolution are not yet implemented.

**Outer macro parsing** — `CxMacro` enum added to the AST with six variants: `Test`, `Inline`, `Reactive`, `Deprecated(Option<String>)`, `Cfg(String)`, `Unknown(String)`. The `single_outer_macro` combinator parses `#[name]` and `#[name(args)]` syntax. The `outer_macros` combinator collects zero or more macros before a declaration. `macros: Vec<CxMacro>` added to `Stmt::FuncDef`.

**Outer macro semantic validation** — Five rules enforced:
- `#[test]` with a return type is rejected (test functions must be void)
- `#[reactive]` produces a locked error: "reserved, post-v0.1"
- `#[cfg]` produces a locked error: "reserved, post-v0.1"
- Unknown macro names are rejected with a locked diagnostic
- `#[test]`, `#[inline]`, and `#[deprecated]` are accepted on functions

## Decisions and direction

**Import syntax is real, resolution is not.** The `#![imports]` block can now be parsed and semantically validated — duplicate aliases are caught, invalid registry paths are rejected. But no actual file loading happens. This means the syntax is locked and tested, but multi-file programs still cannot run. The commit message and the test files confirm this is intentional: the syntax layer ships first, the resolution layer comes next.

**Outer macros are a closed set in v0.1.** The `CxMacro` enum has exactly six variants. Unknown names are rejected outright — there is no user-extensible macro system at 0.1. `#[reactive]` and `#[cfg]` are explicitly reserved with locked error messages, establishing them as post-0.1 features without ambiguity.

**Test infrastructure is building up from both sides.** The `#[test]` attribute is now parsed and validated (no return type allowed), but there's no test runner yet. The import system can parse `std/` paths but can't resolve them. These are converging toward a usable test story — `#[test]` functions + `assert` built-ins + a test runner — but each piece is landing independently.
