## Decisions and direction

**ABI design is now being documented before implementation.** The `cx_abi_v0.1.md` document takes a design-first approach — locking decisions with tests before writing codegen. This is a shift from the frontend development pattern where implementation often preceded documentation. The ABI doc explicitly separates LOCKED decisions (with tests) from OPEN questions (needing design work).

**Scalar layout follows natural alignment.** Every type's alignment equals its size (t8 aligns to 1, t16 to 2, etc.). This is the simplest correct choice for x86-64 and matches what C compilers do. No packed or special alignment modes at 0.1.

**i128 is explicitly noted as emulated on Cranelift.** The ABI doc flags that Cranelift represents i128 as two i64 values, unlike LLVM which treats it natively. This is documented as a known difference that affects performance but not correctness.

**TBool backend representation is an open question.** The three-state value (true/false/unknown as 0/1/2) has its wire format locked from the language spec, but the backend representation — whether it's a u8, an enum, or a tagged union in IR — is explicitly unresolved. This will matter when TBool values cross function call boundaries.
