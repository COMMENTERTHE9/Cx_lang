## Decisions and direction

**String concatenation is type-strict by design.** The semantic layer rejects `str + non-str` at compile time and directs users to string interpolation (`"v{n}"`) instead of implicit conversion. This is a deliberate language choice — Cx won't silently coerce integers to strings in operator context, matching the explicit-over-implicit philosophy already visible in the TBool and unknown systems. The error message itself serves as documentation: "cannot concatenate `str` and `t32` with `+` — use string interpolation."

**`len` returns byte count, not character count.** `len("é")` returns 2, not 1. This is documented in the README and tested explicitly. The decision aligns with Cx's systems-language positioning — byte-level control is primary, and a future `char_count` or grapheme-level API would be separate. The test fixture `t_len_bytes.cx` nails down this contract.

**The `apply_op` signature changed to `&mut self`.** String concatenation requires arena allocation during binary-op evaluation. This is a structural change that ripples through all callers of `apply_op` in `eval.rs` and `exec.rs`. The change was necessary because concatenation interns the result into the string arena. This means the arena grows with each concatenation — relevant to the roadmap's noted limitation that the string arena grows monotonically.
