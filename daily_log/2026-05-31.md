# Cx Daily Log — 2026-05-31

## Snapshot

The biggest single-day tracker sprint so far: 9 commits closed 9 tracker items on submain, covering type-checking hardening, diagnostics quality, a parser UX fix, and a major runtime restructure. The runtime module was split from a single 1700-line file into 6 focused submodules, and the vestigial per-frame arena was deleted entirely. On the semantic side, integer literals are now range-checked at type-check time (not just at runtime), catching overflow errors before execution. Main remains at 182/182 matrix; submain has grown to 198 fixtures across 19 non-merge commits ahead of main.

## Landed today

All 9 commits landed on `origin/submain`. None have been merged to main yet.

- **reject out-of-range integer literals at type-check** (`a63ca2e`, tracker #028) — CONFIRMED. The semantic analyzer now checks that integer literals fit within the target type's range *before* the program runs. A `t8` variable assigned `999` is caught at compile time with a clear error. The check applies to typed assignments, return expressions, struct field assignments, and array elements. Six existing test fixtures were updated to expect the new compile-time error, and two new struct-overflow fixtures were added.

- **reject negative out-of-range literals** (`f60e275`, tracker #037) — CONFIRMED. Extends #028 to cover unary-negated literals. `-200` assigned to a `t8` (range -128..127) is now caught at type-check. New test fixture `t_neg_literal_overflow_t8.cx` added.

- **split runtime.rs into focused modules** (`fa4878a`, tracker #011) — CONFIRMED. The monolithic `src/runtime/runtime.rs` (1725 lines removed) was decomposed into 6 submodules: `scope.rs` (385 lines — scope management, variable lookup), `ops.rs` (249 lines — binary/comparison operators), `eval.rs` (151 lines — expression evaluation), `exec.rs` (532 lines — statement execution), `call.rs` (370 lines — function/builtin dispatch), and `print.rs` (61 lines — output formatting). The core `runtime.rs` now holds only the type definitions and initialization. The old `arena.rs` was also removed as part of this change.

- **delete the vestigial per-frame arena** (`26a4074`, tracker #035) — CONFIRMED. The `Arena` struct and its 64 KB chunk allocation were removed. The arena was a leftover from an earlier memory model that was superseded by `BindingId`-indexed variables (tracker #009). The `ScopeEvent::ArenaReset` diagnostic arm was also removed from `diagnostics.rs`.

- **directive error for `let ... = expr`** (`f0fc976`, tracker #013) — CONFIRMED. The parser now detects when users write `let x = value` or `let x: T = value` and produces a targeted error: *"`let` declares an uninitialized binding and cannot have an initializer — to declare with a value use `x: T = value` (without `let`)"*. Previously this fell through to a raw chumsky parse error. Two new test fixtures cover the untyped and typed cases.

- **real source positions for composite-literal errors** (`967038d`, tracker #031b) — CONFIRMED. `StructInstance` and `ArrayLit` AST nodes now carry source positions. Struct literal analysis gained full field-level checking: unknown fields, missing fields, and type mismatches all report against the struct literal's position rather than position 0. Four new expected-fail test fixtures cover the struct error paths.

- **array element mismatch reports declared type** (`de85b0b`, tracker #031a) — CONFIRMED. When a typed array like `[3: t8]` has a mismatched element, the error now names the *declared* element type rather than inferring from element[0]. The `Analyzer` gained an `expected_array_elem` field that threads the declared type through to the `ArrayLit` analysis arm. One new test fixture.

- **clear error for integer literals exceeding i128** (`d9fcfcc`, tracker #016) — CONFIRMED. The lexer error path now detects all-digit tokens that failed parsing (i.e., overflow beyond `i128::MAX`) and reports *"integer literal out of range for i128"* instead of the generic *"unrecognized token"*. New test fixture `t_int_literal_overflow_i128.cx`.

- **reserve `exit` in the IR validator gate** (`be4affe`, tracker #033) — CONFIRMED. The `exit` builtin was added to the reserved-name list in `src/frontend/builtins.rs`, preventing the IR validator from rejecting programs that call `exit()`. Single-line change.

## Decisions and direction

Today's work shows two clear technical directions:

**Shift-left on error quality.** Five of today's 9 commits improve error messages or catch errors earlier in the pipeline. Integer overflow is now a compile-time error (#028, #037). The `let ... = expr` parse error (#013) guides users toward Cx's actual syntax. Composite literals get real source positions (#031a, #031b). The i128 overflow lexer error (#016) replaces a misleading message. This is systematic work to make the compiler a better teacher.

**Runtime architecture cleanup.** The other 4 commits clean up the interpreter's internal structure. The runtime split (#011) brings the module from an unmaintainable monolith to a clearly organized set of concerns. The arena deletion (#035) removes dead infrastructure. These changes don't affect language behavior but substantially improve maintainability for future work.

The submain-to-main gap (19 commits, 77 files) is the largest it has been. The work is coherent and well-tested (198 matrix fixtures on submain vs 182 on main), but the integration risk grows with each day of divergence.
