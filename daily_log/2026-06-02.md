## Landed today

- **`unknown`-as-value points to `?` literal** (`cc7812a`, tracker #039, `origin/submain`) — CONFIRMED. Writing `unknown` in value position (e.g., `b: bool = unknown`) used to fail deep in the statement parser and report at the `:` token with a ~20-token "expected" dump that never mentioned `?`. The fix adds a `KeywordUnknown` alternative to the primary expression parser that `validate`-emits a targeted error at the `unknown` span: "`unknown` is a pattern keyword, not a value — use `?` for the unknown literal." It recovers as `AstValue::Unknown` so the diagnostic points at the offending token instead of cascading. The emitted error still correctly fails the parse. The `?` literal path and `unknown` as a `when` pattern are unaffected (separate parser branches). Two new test fixtures added: `t_unknown_as_value.cx` and `t_unknown_as_value.cx.expected_fail`. 18 lines changed in `src/frontend/parser.rs`. Commit gates report 243/0 unit tests, 418/0 total, matrix 208/0, parity 134/74/0 on submain.
