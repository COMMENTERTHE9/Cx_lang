## Decisions and direction

**Struct representation as Ptr.** The decision to lower `SemanticType::Struct` to `IrType::Ptr` is a meaningful architectural choice. Structs in the IR are now pointer-typed values — stack-allocated via Alloca, accessed via Load/Store. This is the standard approach for a lowering pass targeting something like Cranelift or LLVM: structs live in memory, not in registers, and field access will be pointer arithmetic plus load/store. The struct layout computation (from Phase 8) is now reachable from the lowering pass, which means field offset calculation for GEP-like operations is available whenever struct field lowering is implemented.

**Struct registry at the module level.** `build_struct_table` scans all top-level `StructDef` nodes once before any function is lowered. This means the struct layout is computed once and shared across all functions — consistent with how `build_signature_table` already works for function signatures.
