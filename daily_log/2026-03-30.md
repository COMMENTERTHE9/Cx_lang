## Decisions and direction

**UTF-8 strict everywhere is now locked.** This is a meaningful decision because it closes the design space: no Latin-1 fallback, no WTF-8, no byte strings masquerading as str. The choice is opinionated and simple — str is always valid UTF-8, period. This aligns with Rust's model and is the right call for a systems language that wants predictable string behavior.

**Semicolons are now partially optional, with a documented ceiling.** The decision to require semicolons on expression statements is a pragmatic parser limitation, not a language design choice. The commit message and roadmap update are explicit: full semicolon-free syntax needs a newline-aware parser, and that's post-0.1. This is an honest acknowledgment of a design/implementation gap rather than pretending the issue is fully resolved.

**Both commits were doc-heavy relative to code.** The UTF-8 commit is pure documentation. The semicolon commit is one line of parser code plus documentation. This suggests the project is in a spec-tightening phase — locking decisions and documenting limitations rather than shipping new features.
