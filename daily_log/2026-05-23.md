## Landed today

**refactor(frontend): add Type::Void AST variant (f3c2856, merged to main via PR #278).** CONFIRMED. Previously, `void` in function signatures was parsed as an identifier and resolved through the struct-type path, causing a spurious "return type mismatch: expected void, got void" error when a void-returning function ended with a trailing void call (like `print()`). This commit gives `void` first-class representation across four frontend files:

- `ast.rs`: Added `Type::Void` in the primitive cluster alongside T8, T16, Bool, etc.
- `lexer.rs`: Added `#[token("void")] TypeVoid` so `void` is now a reserved keyword, not an identifier.
- `parser.rs`: Added `Token::TypeVoid => Type::Void` to the scalar arm of `type_parser`.
- `semantic.rs`: Removed the `if name == "void"` string-match band-aid; added a direct `Type::Void` arm that maps to `SemanticType::Void`. Two new unit tests verify the round trip at parser and semantic levels.

The merge commit is `2a2e1b2` on main.

**feat(runtime): add exit() builtin for explicit exit code propagation (4d612df, on submain).** CONFIRMED, branch-local. This is a substantial commit touching 25 files across four layers of the compiler:

- *Types:* Added `RuntimeError::Exit(i32)` as a control-flow signal in `types.rs`, alongside `EarlyReturn`, `BreakSignal`, and `ContinueSignal`.
- *Semantic:* Added `exit()` analysis in `semantic.rs` — accepts 0 or 1 argument, arity enforced at analysis time, returns `SemanticType::Void`.
- *Runtime:* Implemented in `runtime.rs` — evaluates the optional integer argument, validates it fits i32, returns the `Exit` signal. Out-of-range values surface as `TypeMismatch`.
- *Main loop:* In normal interpreter mode, the `Exit` signal is caught before error rendering, stdout is flushed (to prevent lost piped output since `process::exit` skips Drop), and `process::exit(code)` is called. In `--test` mode, `exit(0)` counts as PASS and `exit(n!=0)` counts as FAIL, with the test runner continuing to the next fixture.
- *Diagnostics:* Added a never-render arm for `RuntimeError::Exit` as a safety net.
- *Matrix harness:* Extended `run_matrix.sh` with `.expected_exit` file support, allowing fixtures to assert specific exit codes and optionally verify stdout alongside them.
- *Test fixtures:* Seven new verification matrix tests: `t_exit_default` (no-arg defaults to 0), `t_exit_zero_explicit`, `t_exit_nonzero` (exit 3), `t_exit_before_output` (exit before any print), `t_exit_after_partial` (exit after partial output, verifies stdout flush), `t_exit_in_branch` (exit from if-branch), `t_exit_in_function` (exit propagates past function-call boundary).
- *Integration tests:* Two new tests in `diff_harness.rs` verify --test mode behavior: `exit(1)` in a `#[test]` function records as FAIL and the runner continues; `exit(0)` records as PASS and the runner continues.

This commit is on submain and not yet merged to main.

## Decisions and direction

**`void` as a first-class type keyword.** The Type::Void refactor is a deliberate language-model decision. `void` is now a keyword with its own token and AST variant, not an identifier that resolves through the struct/type path. This eliminates edge-case bugs where `void` could shadow or collide with user-defined types, and it makes the parser's type resolution path structurally explicit. This is post-release type-system cleanup — the kind of fix that firms up the language's internal representation without changing user-facing syntax.

**exit() as a control-flow signal, not a process kill.** The design choice to propagate `exit()` as a `RuntimeError::Exit(i32)` signal rather than calling `process::exit` directly at the call site is architecturally significant. It means exit() composes correctly with the test runner (where it becomes a test result rather than a process terminator), it allows stdout to be flushed before termination, and it keeps the runtime's control-flow model consistent — `Exit` is a sibling of `EarlyReturn`, `BreakSignal`, and `ContinueSignal`, all carried as `RuntimeError` variants. The --test mode integration is especially important: without it, a `#[test]` function calling `exit(1)` would kill the entire test suite silently.
