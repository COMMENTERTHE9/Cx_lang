## Test runner

The test runner landed as commit `762b188` on `submain`. It adds `assert(cond)` and `assert_eq(a, b)` as built-in functions, plus a `--test` CLI mode. Changes touch eight existing files and introduce four new matrix tests.

The runtime behavior is straightforward: `assert` halts with "assertion failed" when the condition is false, and `assert_eq` halts with an expected/got diagnostic on mismatch. The semantic layer handles recognition and type-checking of both calls. `--test` flag wiring lives in `main.rs`.

The four new tests cover the basics:
- `t77` and `t78` verify that `assert(true)` and `assert_eq` on strings work correctly
- `t79` and `t80` confirm that `assert(false)` and mismatched `assert_eq` fail as expected

This closes the "Basic test runner" hard blocker cleanly.

## Result\<T\> error model

This is the bigger piece of work -- 162 lines of changes across 9 source files, plus 16 new untracked test files (8 test cases with their expected output/fail sidecars). It is not committed yet, but the implementation is complete and all tests pass locally.

The implementation follows the frozen spec exactly: `Result<T>` type syntax with angle-bracket delimiters, `Ok(val)` and `Err("msg")` as expression-level constructors, and `?` as a postfix propagation operator.

The work spans every layer:

**Lexer** -- `Result`, `Ok`, `Err` are now keywords.

**Parser** -- `Result<T>` parses with `<`/`>` delimiters. `Ok(expr)` and `Err(expr)` parse as primary expressions. `?` parses as a postfix operator after index expressions.

**Semantic layer** -- This is where most of the interesting decisions live. `Ok(expr)` wraps the inner type in `Result<T>`. `Err(expr)` enforces a string argument and produces `Result<Unknown>`, since the semantic layer can't know what `T` should be at an `Err` site without bidirectional type inference. The `?` operator validates that the inner expression is `Result<T>`, checks that the enclosing function returns `Result<U>`, and unwraps to `T`. The existing `Unknown`/compatibility rules handle the `Result<Unknown>` case naturally.

**Runtime** -- `Value::ResultOk` and `Value::ResultErr` variants added. The `?` operator evaluates its inner expression: `Ok` unwraps, `Err` triggers `EarlyReturn(ResultErr(...))`. This reuses the existing `RuntimeError::EarlyReturn` path that was already in place for function returns, which keeps the implementation clean.

**IR** -- Stub entries only. `ResultOk`, `ResultErr`, `Try`, and the `Result` type are all `unsupported!` in the lowerer. This is interpreter-only for now, consistent with how structs, methods, and `when` expressions are handled.

The eight new tests cover the important cases:
- `t81`-`t84` verify `Ok` returns, `Err` returns, `?` unwrapping, and three-function `?` propagation chains
- `t85`-`t88` verify rejection of `?` outside Result-returning functions, `?` on non-Result values, Result value comparison with `assert_eq`, and that `Err(42)` is rejected (must be a string)

The design decision to keep `Err` string-only at 0.1 is worth noting. No typed error payloads yet. This keeps the type system simpler and avoids a whole category of inference problems that would otherwise need solving before the milestone.
