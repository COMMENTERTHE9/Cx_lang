## Loop, break, continue in the IR

The interesting change is `bcd49f2` on `origin/submain`: `src/ir/lower.rs`, +215/−16. It threads a `LoopContext` (header id, exit id, and the ordered set of loop-carried bindings) through statement lowering so that structured jumps have somewhere to go.

Concretely:

- `lower_loop` is new. It builds the CFG for an infinite loop: entry jumps unconditionally into the header, the header runs the body, and the exit block is constructed with block params matching the ordered binding set.
- `lower_while` was updated in the same pass. The exit block now gets proper block params, and the header's `Branch` terminator passes real `else_args` into it instead of an empty vector. This is the bug fix that lets loop-carried SSA actually survive the while's early-exit edge. The previous version was emitting a terminator that referenced a block it had not wired up.
- `SemanticStmt::Break` lowers by gathering the current SSA values for every ordered binding, emitting `IrTerminator::Jump { target: exit_id, args: exit_args }`, and sealing the block.
- `SemanticStmt::Continue` does the same thing against `header_id` instead of `exit_id`.
- `loop_ctx: Option<&LoopContext>` is now plumbed through `lower_stmt`, `lower_if_else`, and `lower_if_chain`. Previously `lower_if_chain` hardcoded `None` when lowering its inner sequence, which meant a `break` inside a nested `if` inside a loop would not resolve to the enclosing loop's exit. That is fixed.
- `break`/`continue` outside a loop return `UnsupportedSemanticConstruct`. Missing bindings in the SSA env return `InternalInvariantViolation`.
- The three existing "rejects_unsupported_statement" tests were retargeted from `SemanticStmt::Loop` to `SemanticStmt::For`, which is the next statement still rejected by the lowering pass.

One incidental tell in the diff: `SemanticExprKind::ResultOk`, `ResultErr`, `Try`, and `SemanticType::Result(_)` each got an `unsupported!` arm so the match stays exhaustive. The semantic layer has already grown Result shapes on submain. Lowering has not implemented them yet. The arms are placeholder rejections, not real handling.

This commits the backend to structured `break`/`continue` — no labeled breaks, no early-return tricks. That matches the roadmap, where "labeled breaks for nested loops" is a Post-0.1 item.

## Agent operating doctrine

The other commit is `03d2da6`: a new `docs/AGENT_OPERATING_DOCTRINE.md`, +283 lines. It defines a six-phase workflow (Recon, Design Lock, Predicted Fallout, Implementation, Validation, and one more phase later in the file) and the notion of "task packets" — one logical unit of work per prompt instead of one file per prompt. This is the first process doc of its kind in `docs/`, and its existence is the decision. Prior work had no written protocol.
