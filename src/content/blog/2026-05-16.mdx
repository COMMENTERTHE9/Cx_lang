## The CX-218 fix: SemanticType::Numeric in lower_binary

When the lowering pass encounters an arithmetic expression like `0 + 0` or `3 * 4`, the result type can be `SemanticType::Numeric` -- a placeholder that means "some numeric type, not yet resolved." The `lower_value` path already handled this by resolving Numeric to the target's `numeric_literal_ir_type()` (I64 on 64-bit), but `lower_binary` didn't. It called `lower_type(result_ty)` directly, which returned `UnsupportedSemanticType`, and every `assert_eq(arith_expr, literal)` test case silently exited with code 127 (SKIP) instead of actually running.

The fix mirrors what `lower_value` already does: resolve `SemanticType::Numeric` to `ctx.target.numeric_literal_ir_type()` in the arithmetic arm of `lower_binary`. This is a pragmatic duplication rather than a centralized resolution -- given the current lowering architecture, each lowering path resolves the placeholder independently. More Numeric-resolution fixes in other lowering paths wouldn't be surprising as parity coverage expands.

The net effect: arithmetic JIT parity tests that were invisible (SKIP) are now running and passing. Zero PARITY_FAIL across the board.
